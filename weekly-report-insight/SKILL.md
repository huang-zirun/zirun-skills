# 周报想法记录生成 Skill

## 核心任务

帮助用户撰写周报中的"想法记录"部分，从具体工作场景中提炼出有深度的认知方法论。

## 输出风格要求

### 1. 聚焦具体场景
- 必须从用户本周实际工作内容中选取**一个具体场景**
- 避免泛泛而谈，深入挖掘一个点的认知价值
- 场景可以是：排查问题、开发功能、会议讨论、面试评估、技术对接等

### 2. 认知方法论风格
- 采用哲学思辨的表达方式
- 适当引用哲学概念（如康德、波普尔、维特根斯坦等）
- 使用生动比喻开篇（如"排查问题就像是在黑暗中摸索开关"）
- 从具体经验提炼普适性认知

### 3. 结构要求
- **一段连贯文字**，不要分多段
- 用逗号、破折号、分号连接句子，保持流畅
- 结构：开篇比喻 → 具体经历 → 认知转变 → 迁移应用 → 结尾金句

### 4. 语言特点
- 简洁有力，短句为主
- 层层递进，逻辑清晰
- 结尾必须有一句**加粗的金句**升华主题

## 执行流程

1. **分析工作内容**：从用户提供的周报工作内容中，识别最有认知价值的具体场景
2. **确定核心洞察**：提炼这个场景背后的认知方法论（如提问方式、思维方式、工作方法等）
3. **撰写想法记录**：
   - 用比喻开篇
   - 叙述具体经历
   - 引出认知转变
   - 迁移到其他场景
   - 金句结尾
4. **检查格式**：确保是一段连贯文字，结尾有加粗金句

## 示例

**输入（工作内容）：**
- 协助Reed排查OpenClaw授权失败问题
- 开发BOSS直聘自动打招呼脚本
- 修复OCR API调用超时问题

**输出（想法记录）：**

排查问题就像是在黑暗中摸索开关，本周协助Reed解决OpenClaw授权失败时，我深刻体会到真正困住我们的往往不是问题的复杂度，而是提问的方式——起初我问"为什么授权失败"，这是试图打开一个黑箱，得到的只有沉默，后来我问"在什么条件下可以成功"，局面豁然开朗，换网络、换账号、换版本，每一次尝试都在缩小包围圈，最终发现是特定版本与网络环境的组合触发了验证冲突，问题还是那个问题，但提问方式变了，答案就自己浮现了。这让我意识到排查的本质不是逆向工程，而是实验设计，我们无法直接"看见"问题，只能通过控制变量在可能性空间中逐步排除，就像波普尔说的，科学理论无法被证实，只能被证伪，排查也是如此，我们不是在"找"问题，而是在"排除"不是问题的东西。把这个视角切换到面试评估，道理相通，与其问"这人有什么缺点"，不如问"在什么条件下他能发挥最大价值"，缺点是绝对的存在，条件却是相对的变量，OCR的速率限制不是缺陷只是使用条件，BOSS直聘的反爬不是障碍只是适配条件。**很多时候，所谓问题，不过是我们还没找到与之共处的方式。**

## 注意事项

- 不要罗列多个观点，聚焦一个核心洞察
- 不要使用"首先...其次...最后..."等结构化表达
- 不要写成工作总结或复盘，要提炼认知方法论
- 金句要简洁有力，能够独立传播
